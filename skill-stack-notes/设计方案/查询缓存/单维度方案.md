# 服务缓存优化方案

## 服务缓存存在的问题:
1. favors 列表缓存,收藏和订阅是通过类型位图(如000110)生成的组合,且发布时列表未处理组合key(看似自由组合实则写死了).未用到zset,且每页是一个key缓存的整个分页数据
2. 列表分页缓存的是整个res_info数据,占用内存大,无法知道资源变更
3. 历史列表新增和移除数据造成列表数据失效,需要重新缓存
4. 通过page和page_size分页,数据变化会出现漏查或重复(不一致可接受)

## 缓存列表解决方案
### A. 按资源类型缓存数据,服务端合并排序再截取
  - 优缺点: 适合数据量小于5k,主要按时间排序,App通用做法
### B. 按资源类型缓存数据,通过redis做交集
  - 优缺点: 高并发不友好,会生成临时key,占用CPU
### C. 列表全量zset(成员数百万是常见的),服务端做过滤截取或创建子ZSET
  - 优缺点: 简单实用可控
### 分页优化解决方案
> 向后兼容支持cursor.ZSET的ZRANGE有深分页性能问题,高并发更适合score/cursor查询

- 有cursor 参数直接解析
  - 规则: id+score
- ~~没有cursor参数,将page/page_size换算为cursor,并短期缓存.~~
  ```
  Key: page_cursor:{uid}:{scene}:{page}:{page_size}
  TTL: 1~5 分钟
  Value: cursor
  ```
  - scene取值
    - collect_all|collect_:res_type(res_type目前有article|ticker|video|live|dynamic 可按资源自由组合)
    - fans_all|subscribe_all|subscribe_subscription
    - history_all|history_:res_type(res_type目前有article|ticker|video|live|dynamic 可按资源自由组合)

## C方案优化 -> 最终方案
> 选择方案原因: 用户数据天然非热点key且有限,且有组合类型查询需求,非video类型密度容易稀疏.

- 有子类型 ZSET 则直接查询(返回处理后的{type,id,score}数组)
- 有全量ZSET则用ZSCAN重建子ZSET,再用ZRANGE取数据.(过期时间为30天,小于2天则重置过期时间)
  - 设置子类型列表 TTL 24*7小时
  - 并维护子类型的反向索引集合
- 没有全量ZSET,需要查询数据库page固定为1查询数据并返回(保证用户体验,必须有返回数据,并防止page过大的深分页问题).作为索引无需TTL
  - 通过其他手段(限长+主动剪裁/清理)处理
  - 必须异步查询所有数据并缓存全量ZSET
  - 需标记防止重复同步
  - 需标记空数据防止重复
  - 主体 key 格式: `version:service:scene:res_type:res_id`
  - 成员格式 `${type}_${id}` (ZSCAN~~前缀匹配性能极佳~~华为云不支持^和$,使用sub方法)
### 安全防御事项
- zscan根据zcard设置batch(返回值非精确)的大小(500<10万<1000),如果子 ZSET 是多类型组合,~~这需要多次扫描或~~执行lua脚本过滤
  - 百万级数据batch 1000次大概 600ms
  - 相同 lua 脚本加载 sha 相同(限制 lua 内存占用,删除缓存: `SCRIPT FLUSH`)
  - MySQL查询的循环中,不能单靠时间,还得做id判断,最后加个循环次数或id判断结束
- 发布资源时,根据反向索引集合判断子ZSET是否存在(要对key做类型过滤).存在则做增量同步,不存在可以删除反向索引集合的key.子zset的构建由下次请求触发
